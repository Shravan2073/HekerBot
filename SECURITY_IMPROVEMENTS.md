## 2. Security Improvements for HekerBOT

To ensure HekerBOT runs securely without compromising the host machine, the following security hardening measures should be implemented:

### A. Execution Isolation (Docker)
Currently, if the agent runs on the local host, any malicious command generated by a hallucinating LLM could destroy your host system. 
- **Action:** Always enforce **Docker Mode**. 
- **Improvement:** Configure the Docker container with `--cap-add=NET_RAW` so that tools like `masscan` and `nmap` can send raw packets without needing full `root` or `sudo` access on the host system.

### B. Network Segmentation
- **Action:** When scanning targets, ensure the HekerBOT Docker container is placed on a dedicated bridged or macvlan network.
- **Improvement:** This prevents the AI from accidentally discovering and attacking internal infrastructure (like your router or other personal devices) unless explicitly authorized.

### C. Secrets Management
- **Action:** API keys (OpenAI, Anthropic, Gemini, OpenCode) are currently stored in `.env`.
- **Improvement:** Ensure `.env` is firmly included in `.gitignore`. For production deployments, consider integrating a secrets manager (like HashiCorp Vault) or using Docker Secrets so keys are never exposed in plaintext to the AI agent's environment.

### D. Command Allowlisting
- **Action:** The agent currently has the theoretical ability to execute any shell command.
- **Improvement:** Implement a strict command validator in `executor.py` that only permits specific binaries (e.g., `nmap`, `ffuf`, `masscan`, `ping`) and flags dangerous flags (like `rm -rf` or `> /etc/passwd`).

### E. LLM Fallback & Redundancy
- **Action:** The mission aborted because a single API endpoint failed.
- **Improvement:** Implement a fallback mechanism in `brain.py` that automatically switches to a backup model (like `gemini-2.5-flash` or a local Ollama model) if the primary API endpoint returns a 404, 502, or invalid JSON.
